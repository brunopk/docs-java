# OAuth

## Authorization code flow with PKCE

The following instructions describe part of a custom implementation of the **authorization code flow with PKCE** for the [mis-gastos-backend](https://github.com/brunopk/mis-gastos-backend) project using only some of the libraries provided by Spring for OAuth2 and **Google** as provider. **Most documentation and tutorials recommend delegating configuration and OAuth2 concerns—such as token generation and validation—to these libraries** instead of implementing them manually as shown here.

- **Implement and endpoint to receive **authorization code** and code verifier (PKCE)** in order to exchange them for the Google bearer token :
  - Refer to [line 23](https://github.com/brunopk/mis-gastos-backend/blob/e1c1fb5418fa7c173df9886f7e80ba93d45583c6/src/main/java/com/bruno/misgastos/controllers/OAuth2Controller.java#L23) in [`OAuth2Controller`](https://github.com/brunopk/mis-gastos-backend/blob/e1c1fb5418fa7c173df9886f7e80ba93d45583c6/src/main/java/com/bruno/misgastos/controllers/OAuth2Controller.java) for an example of this endpoint.
  - The bearer token is obtained in [line 41](https://github.com/brunopk/mis-gastos-backend/blob/e1c1fb5418fa7c173df9886f7e80ba93d45583c6/src/main/java/com/bruno/misgastos/services/OAuth2ServiceImpl.java#L41) of [`OAuth2Service`](https://github.com/brunopk/mis-gastos-backend/blob/e1c1fb5418fa7c173df9886f7e80ba93d45583c6/src/main/java/com/bruno/misgastos/services/OAuth2ServiceImpl.java).
  - This token is validated in [line 42](https://github.com/brunopk/mis-gastos-backend/blob/e1c1fb5418fa7c173df9886f7e80ba93d45583c6/src/main/java/com/bruno/misgastos/services/OAuth2ServiceImpl.java#L42) of [`OAuth2ServiceImpl`](https://github.com/brunopk/mis-gastos-backend/blob/e1c1fb5418fa7c173df9886f7e80ba93d45583c6/src/main/java/com/bruno/misgastos/services/OAuth2ServiceImpl.java).
  - After validating the token, a `UsernamePasswordAuthenticationToken` instance will be set on the security context of the request in order for Spring internal security filters to verify that the user was authenticated (take into account that manually creating and setting an instance of `Authentication` interface like this, is something discouraged in most documentations and tutorials).
- Refer to the [Login](https://github.com/brunopk/mis-gastos-web/blob/0a22debfd813fb7fa6bca91f2391be8405b9576e/src/components/pages/login/Login.tsx) component in [mis-gastos-web](https://github.com/brunopk/mis-gastos-web) to understand how frontend starts the authorization code flow and how it obtains the bearer token invoking the callback request with authorization code and code verifier.

  - The authorization code flow (only frontend part) can be summarized into these steps:
  
    1. Start authorization with Google ([line 72 of `Login.tsx`](https://github.com/brunopk/mis-gastos-web/blob/0a22debfd813fb7fa6bca91f2391be8405b9576e/src/components/pages/login/Login.tsx#L72)).
    2. Once the Login component detects that Google returned to the login URL with authorization code in query params, invoke the callback endpoint in mis-gastos-backend ([line 83 of `Login.tsx`](https://github.com/brunopk/mis-gastos-web/blob/0a22debfd813fb7fa6bca91f2391be8405b9576e/src/components/pages/login/Login.tsx#L83)).

  - **Code verifier is generated by the frontend and saved into local storage** ([line 70 in `Login.tsx`](https://github.com/brunopk/mis-gastos-web/blob/0a22debfd813fb7fa6bca91f2391be8405b9576e/src/components/pages/login/Login.tsx#L70)).
